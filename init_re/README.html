Data overview

The datasets represent three different formats of the following data:

  • Global wind turbines clustered using a neighbourhood distance of 800m;
  • Global solar installations clustered using a neighbourhood distance of 400m.

The formats comprise:

  1. ESRI geodatabase (*.gdb): proprietary geospatial file format for use with ArcGIS that contains two layers in the same file, one for solar and one for wind;
  2. Geopackages (*.gpkg): open source geospatial file format that can work with a variety of software and can store different geometries natively;
  3. Comma-delimited files (*.csv): the simplest format, these files represent the centroid of each data record, i.e.Â point data, as X and Y coordinates in a data table.

Also included is an archive folder, analysis, that contains four R scripts and accompanying data used for acquiring and processing the data.

The datasets are all provided in Eckert IV equal-area projection (EPSG:54012).

Solar (global_solar_2020.*): 35,272 observations

Please note that, as described in the accompanying paper, these datasets contain ever single observation, including sole turbines and solar panels. This is for the sake of completeness; to work with the data, we recommend filtering the data as per the power estimation analysis in the accompanying paper (Methodology): wind installations with more than four turbines and not in a cell identified as urban or water (e.g.Â dplyr::filter(turbines>4 & urban!=1 & water!=1)), and solar panels not in an urban cell and over 1 hectare in panel area.

The analysis folder contains four R scripts and seven data files. The package pacman, installed at the start of the script with the code if (!require(pacman)) install.packages('pacman'), will install the necessary packages not currently installed on your machine and attach them when installed. The scripts perform spatial operations and as such require copies of GDAL and GEOS to be present on your machine.

All file paths are relative, meaning they will only work if the script is running in the correct working directory. This can be set by running the command setwd(your_path_to_analysis_folder).

While these scripts have been written to be as user-friendly as possible, they do require some supervision. For example, 02_osm_processing.R imports the raw downloaded OSM data (list.files(path='data', pattern='solar_raw')) assuming that the user has not changed the file name and location after running the previous script.

The first two scripts take some computing power and time: they were originally performed on a desktop PC with i7-6700 processor and 32GB RAM. The first script relies on continually querying the OSM data servers until a download slot is available as access to OSM data is rate limited. This is entirely dependent on server traffic at the time the script is run; the last run to generate the data presented here took just over four hours. The second script processes large spatial files; the last time this was run it ran for just over half an hour.
